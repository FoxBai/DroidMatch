## 2026-06-26

| Decision | Rationale |
|---|---|
| Project name is DroidMatch | Establish a new identity independent from HandShaker and Smartisan. |
| Build a modern replacement, not a clone | Preserve valuable workflows while avoiding old brand, UI assets, and binary implementation. |

| M1 ADB download uses receiver-paced chunks first | The Mac harness ACKs each chunk before Android reads and sends the next one, proving multi-chunk correctness without introducing multi-stream scheduling before the real-device matrix. |
| M1 ADB download may fill a small per-stream window | Once basic receiver-paced correctness was proven, Android can keep up to 4 chunks or 2 MiB in flight on a single download stream. This preserves ACK backpressure while removing the per-chunk round-trip bottleneck seen on Slot D. |
| M1 resume starts with sidecar fingerprint validation | The Mac harness stores Android's accepted source fingerprint next to the partial file and sends it with non-zero offset requests, letting Android reject stale resumes before transfer scheduling is added. |

| App-sandbox upload resume may truncate duplicate partial bytes | If Android wrote a chunk but Mac missed the ACK, the next app-sandbox resume should roll back to Mac's requested offset and replay rather than fail on a longer partial file. |

| Reconnection authentication is a second challenge-response step after ServerHello | Client and server HMAC proofs cover both fresh nonces with role separation, preventing replay that a ClientHello-only proof cannot prevent. |

## 2026-07-14

| Decision | Rationale |
|---|---|
| App-sandbox upload never downgrades an unsupported atomic replacement | The final ACK is a commitment that the completed hidden partial replaced the destination as one filesystem operation. Falling back from `ATOMIC_MOVE` to a normal replacing move could expose a missing or partial destination after failure. Android now fails before final ACK, retains the resumable partial, and leaves the prior destination untouched when atomic replacement is unavailable. |
| App-sandbox recursive deletion never follows symbolic directories | M1 has no symbolic-link wire kind, so publishing one as a normal file or directory creates a non-canonical path. More importantly, Java `File.isDirectory()` and `listFiles()` follow a directory link and could make recursive deletion leave the app-owned root. Listings now omit links, while recursive cleanup unlinks the link node as a leaf and preserves every target entry. |
| App-sandbox upload partials are opened through one no-follow channel | The predictable hidden resume name is durable provider state, but preflight with `File.isFile()` followed by `RandomAccessFile`/`FileOutputStream` can follow a substituted symlink and truncate or overwrite its target. Fresh creation now uses delete plus `CREATE_NEW`; resume requires a no-follow regular file and performs size validation, ACK-loss truncation, positioning, and append on one descriptor. |
| App-sandbox final ACK requires forced payload data before atomic replacement | `flush()` and `close()` release Java resources but do not prove the completed partial reached stable storage. Android now calls `FileChannel.force(true)` on the same no-follow descriptor, closes it, and only then attempts the same-filesystem atomic replacement. A synchronization or atomic-move failure retains the partial, preserves the prior destination, and fails before final ACK. |
| App-sandbox download identity comes from the opened descriptor | Reading `File.length()`/`lastModified()` after open can bind transfer bytes to one inode while fingerprint metadata comes from a replacement path. Path + size + mtime also accepts an atomic replacement when its visible metadata is preserved. Product download open now uses `fstat` on the exact `FileDescriptor`; device/inode/ctime are hashed with the logical path into the opaque etag, while size/mtime come from the same stat. This detects replacement without an O(file-size) SHA-256 pass before first byte. |

## 2026-07-16

| Decision | Rationale |
|---|---|
| Resumable Mac uploads require format-v2 strong source identity | Size and millisecond mtime cannot distinguish a rapid atomic replacement. One attempt-long `O_NOFOLLOW` descriptor plus size, nanosecond mtime/ctime, filesystem, and inode binds every read and checkpoint to the same object. Non-zero v1 state is rejected before reconnect instead of risking a mixed remote file. |
| A local download namespace has one non-terminal owner and a crash-detectable commit | Final, partial, sidecar, sidecar recovery entries, fixed commit marker, and fixed replaced entry conflict as one lexical namespace; product execution adds a parent-inode/case-aware in-process registry, sorted cross-process advisory locks, security-scope lease, and directory FD. A verified parent-relative private root/anchor plus persistent empty SHA-256-named lock files keep cooperating processes on the same lock inodes without directly storing paths. Fresh work locks the partial before sidecar removal and resets that FD before connecting. Publication synchronizes a `0600` marker, uses `RENAME_EXCL` or validated `RENAME_SWAP`, and retains the displaced old target until sidecar cleanup succeeds. Finalization then removes the old target, synchronizes the directory, and removes the marker; earlier failure restores old target/candidate with the marker retained, republishes the checkpoint, and only then retires the marker. A failed restore remains interrupted; unprovable rollback reports non-retryable `commitUncertain`. Required sync improves process-crash recovery without claiming complete power-loss or malicious same-UID resistance; persistent hashes are pseudonymous metadata, not encryption. |
| Private recovery files use exact pinned-parent single-link operations | Download/upload sidecars, queue manifests, and bookmark archives are untrusted directory entries. Fixed `.<name>.pending`/`.<name>.removing` nodes make interruption discoverable; complete stat, parent-rebinding checks, EXCL/SWAP validation, and file plus directory `fsync` are required. Every used parent permanently retains one fixed zero-byte `0600` `.droidmatch-private-atomic-lock`; no-follow owner/type/link/mode plus named-inode checks around exclusive `flock` serialize legitimate same-process and cross-process read/save/remove calls without exposing target names. Unsafe lock nodes and crash markers remain fail closed, while a malicious same-UID process can still ignore advisory locking and race the final full-stat-to-unlink interval. Failure proves rollback or reports `commitUncertain` with the scene preserved. |
| Scheduler restoration requires a strictly incomplete known-total checkpoint | A checkpoint at its known total may represent a transfer that committed before cleanup; `0 / 0`, unknown total, or conflicting totals are equally ambiguous. Only structurally/path-valid `0 <= offset < total` state restores paused. Upload-v2 source identity is compared only after AppSupport grants the bookmark lease, but still before the coordinator creates a client. All other active state becomes non-replayable `interrupted`. |
| Product persistence repair re-enters the complete authorization transaction | A repaired manifest cannot be canonicalized from path strings alone. While execution remains held, AppSupport reloads bookmarks, acquires every checkpoint security scope and pinned download context, submits one immutable restore plan to Core, verifies all required targets, and only then activates. Any failure reasserts reload-required, so retry never publishes a partially authorized queue. Session suspension similarly delays settlement of unsafe executors until unwind; only an irreversible locally committed download can truthfully replace the conservative interrupted state. |
| Exact Android directory queries have separate retrieval and scan horizons | One page remains at most 1,000 and query tokens stop at 10,000 retrievable entries. If a provider proves more rows remain at the last admissible window, the response is error-only `unsupportedCapability`, never an empty token that would silently claim completeness. App Sandbox/SAF inspect at most 25,000 matching or filtered rows and use the same stable error when exact evaluation would exceed bounded work; MediaStore keeps provider-pushed pagination. |
| Local App and DMG publication use recoverable stable transactions | Same-filesystem candidates and explicit owner/state markers allow the next invocation to recover tested process termination between atomic replacements and bookkeeping. Owner identity combines PID, boot session, and process start time, preventing a reused live PID from pinning stale state. App first publication uses `RENAME_EXCL`; existing-App replacement uses identity-checked `RENAME_SWAP`. DMG initialization first synchronizes owner, a root/parent-bound marker, and `building` state in a private process-instance-scoped sibling, then publishes the complete stable transaction with `RENAME_EXCL`; dead strictly allowlisted and legacy partial initializers recover, while active, forged, or unknown layouts fail closed. DMG absence uses EXCL, replacement uses two-way-checked SWAP, and rollback uses EXCL/SWAP according to recorded prior state. Recovery binds previous/candidate/canonical nodes before and after transitions by device, inode, size, and SHA-256; tests cover every initialization boundary, live-PID/stale-start recovery, a real building hard kill, and preserve/fail closed on concurrent insertion or replacement. Release readiness also rebinds its result to the same local HEAD and clean worktree after slow checks. These `SIGKILL` regressions are not a power-loss durability or release-signing claim. |
| Tooling shares the source reviewability ceiling | The original size gate covered only Swift/Java/Kotlin, leaving release/evidence shell and Python scripts unreported even at 773–800 lines and allowing the physical-device orchestrator to reach 3,277 lines. All `tools/**/*.sh` and `tools/**/*.py` files now share the 800-line default with no exception. The runner is a 673-line final orchestrator over explicit usage, option/validation, device-control, privacy/evidence, App Sandbox probe, result-log, and cleanup contracts; each helper fits the same default. |

## 2026-07-17

| Decision | Rationale |
|---|---|
| Developer ID signing, notarization, and release automation remain owner-deferred until explicitly reopened | Current local development, self-use, and controlled testing are already served by the verified ad-hoc App/DMG path, so release credentials and Apple submissions should not expand the M1 scope. This does not claim an unsigned build is ready for ordinary distribution: a future direct-download release outside the Mac App Store should still use Developer ID and notarization for the default Gatekeeper path. |
| Fault-proxy permission hooks are self-contained and aggregate-only | A generated hook runs in a fresh process and cannot safely depend on unexported parent shell functions. It executes adb with output suppressed, emits only the numeric command status, and never repeats the private serial, adb path, command arguments, or platform exception text. Offline tests execute generated success and failure hooks so source-text inspection cannot mask a clean-shell failure. |
| Minimize one Mac authentication flow to the necessary Keychain operation | An explicit reconnect selects and loads the exact fingerprint-bound record once, then performs no recency write and passes the proven Core credential to the same-generation retry gate. First pairing atomically add-only publishes its provisional record, treats every duplicate pairing ID as a collision without reading or updating the existing item, then returns that fresh Core record directly to the immediate proof instead of reading it back. The coordinator clears its reference after lifecycle ownership is installed and releases any pre-assembly reference on disconnect or keepalive failure. Fingerprint revalidation, rollback, invalidatable gate teardown, and Presentation/log/diagnostic/persistence exclusion remain intact. |
| A running local App detects transactional replacement instead of pretending to update in place | `RENAME_SWAP` safely publishes new bytes but cannot change the executable already mapped into a running process. AppSupport therefore captures the vnode backing dyld image zero through `proc_pidinfo` and polls the published path, closing the launch-to-monitor path race. Replacement, removal, or a non-regular node irreversibly stops discovery, uses the existing ordered session disconnect, disables stale UI, and exposes only a localized Quit action; one process-owned window lease set prevents a closing window from stopping another. No Keychain read, auto-relaunch, revision-string guess, or release-signing claim is introduced. |

| Decision | Rationale |

| Device marketing names remain canonical catalog identities rather than machine-translated UI prose | The public Google Play device catalog supplies one untagged `Marketing Name` text field per row, with no locale-tagged field or image-asset field. The implemented language-aware display selects only a reviewed manufacturer-published, locale-tagged local alias and otherwise falls back to the canonical catalog name, then the sanitized technical model. | Exact device artwork requires a separately licensed and integrity-bounded asset manifest and loader | Search-engine scraping, hotlinking, and per-device image lookups create privacy, availability, copyright, and variant-mismatch risks. Any future exact artwork must bind source and reuse permission, locale/variant identity, content hash, and an allowlisted inert MIME type; loading must enforce the byte limit and hash before decode, then decoded pixel dimensions and frame count. Until then the product keeps bundled generic form-factor symbols. |
| Attended 704SH layout commands are process-group bounded without guessing test-package ownership | The API 26 OEM installer can publish the test package yet leave `adb install` pending indefinitely, and a real timeout showed that an immediate absent-package query can later be invalidated by a device-side commit. Device queries stop after 15 seconds, cleanup after 30 seconds, and interactive install/instrumentation after a configurable 300 seconds with a 600-second ceiling. Any timed-out create-only install remains ownership-unresolved whether the package is currently visible or absent; it never authorizes product replacement or test-package removal. The operator must wait and recheck rollback/late commit or otherwise establish ownership. This preserves product data while preventing a hung attended diagnostic from occupying the runner forever. |
| Android main-screen prose disables automatic hyphenation across the supported API range | A real 704SH screenshot showed API 26 rendering the unhyphenated English word `system` as `sys- / tem` at 1.3 font scale. Every `TextView` and `Button` owned by the `DroidMatchScreen` main hierarchy now uses the simple line breaker with hyphenation disabled; system-owned dialog views stay outside this policy. Normal words wrap at boundaries, unusually long tokens may still wrap, and the main screen never invents a visible hyphen absent from the localized source string. The exact 704SH profile asserts this hierarchy together with existing height and full-scroll bounds. |
